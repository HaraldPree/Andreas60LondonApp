# v1.10.6 — Expliziter MIME-Type für Share-API (Samsung-Fix)

**Datum**: 23.05.2026
**Status**: deployed
**Git Tag**: v1.10.6

## Anlass

v1.10.5 (Harald-Commit) hat 3 sichtbare Save-Buttons gemacht — UX-Lösung
für den „Teilen-Sheet öffnet sich aber ist nicht da"-Bug. Sauberer
Ansatz, aber der **Share-Button selbst** funktioniert weiterhin nicht auf
Samsung Internet weil unten drunter ein subtiler MIME-Type-Bug versteckt
ist.

## Bug

`navigator.share({files: […]})` braucht eine `File` mit korrektem MIME-Type.
Bisherige Implementation:

```ts
const file = new File([ready.blob], ready.filename, {
  type: ready.blob.type,
});
```

`ready.blob.type` ist auf manchen Engines **leer** (z.B. react-pdf's
`pdf().toBlob()` setzt nicht immer einen MIME) → `file.type` ist leer →
Samsung Internet lehnt die Datei stillschweigend ab → Share-Sheet öffnet
sich nicht.

## Fix

Expliziter MIME-Type beim File-Konstruktor:

| Komponente | Vorher | Nachher |
|---|---|---|
| `PdfBookExportButton.tsx` | `type: ready.blob.type` | `type: "application/pdf"` |
| `PhotoBookExportButton.tsx` | `type: ready.blob.type` | `type: "application/zip"` |

Damit weiß Samsungs Share-API sicher was es hat → akzeptiert die Datei →
Sheet öffnet sich.

## Erwartetes Resultat

| Vorher | Nachher |
|---|---|
| Blauer „PDF speichern"-Button → nichts passiert | Samsung Share-Sheet öffnet sich |
| Sekundär-Button „PDF in neuem Tab" funktioniert | unverändert |
| Tertiär-Button „Direct-Download" funktioniert | unverändert |

Wenn Share weiterhin nicht greift, hat der User immer noch zwei Fallback-
Optionen aus v1.10.5.
