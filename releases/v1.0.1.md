# v1.0.1 — Foto-Sharing Backend-Foundation

**Datum**: 21.05.2026
**Status**: deployed (Backend bereit, Frontend folgt in v1.1.0)
**Git Tag**: v1.0.1
**Bump-Grund**: PATCH — internes Setup ohne sichtbares User-Feature

Erstes von zwei Releases zur **Phase-1 Foto-Sharing**. Dieses Release
liefert die Server-Infrastruktur; die Benutzer-UI kommt in v1.1.0.

## Was ist neu

### Neue Dependencies
- `@vercel/blob` ^2.4.0 — Objekt-Storage für Foto-Bytes
- `@vercel/kv` ^3.0.0 — Redis-kompatibler Key-Value-Store für Metadaten

### Neue Types
- `src/types/sharedPhoto.ts` — `SharedPhoto`, `SharedPhotoVisibility`, `SharedPhotoView`
- 3-Stufen-Sichtbarkeit definiert: `"private" | "celebrant" | "group"`
- Hilfsfunktion `canViewSharedPhoto(photo, viewerName, isCelebrant)`
  zentralisiert die Sichtbarkeits-Logik (eine Wahrheit, kein doppelt-implementieren)

### Neue Libraries
- `src/lib/sharedPhotoStore.ts` — Server-side Blob+KV Helpers:
  - `uploadSharedPhoto(...)` — Voll + Thumb hochladen, KV-Index pflegen
  - `listSharedPhotos(tripSlug)` — alle Fotos einer Reise
  - `getSharedPhoto(id)` — einzelner Record
  - `updateSharedPhotoVisibility(id, newLevel)` — Sichtbarkeit ändern
  - `withdrawSharedPhoto(id)` — vollständige DSGVO-konforme Löschung
  - `isStorageConfigured()` — Feature-Flag-Check für Env-Vars
- `src/lib/consentStorage.ts` — Client-side localStorage Helpers:
  - `readConsent(tripSlug, userName)` — aktueller Status
  - `acceptCurrentVersions(...)` — AGB v1.0 + Datenschutz v1.0 markieren
  - `revokeConsent(...)` — Widerruf
  - `hasFullConsent(state)` — Boolean-Check

### Neue API-Endpoints (alle PIN-geschützt via Middleware)
- `POST /api/photos/share` — Foto + Sichtbarkeit hochladen (multipart/form-data)
- `PATCH /api/photos/share?id=...` — Sichtbarkeit ändern (nur Uploader)
- `DELETE /api/photos/share?id=...&requesterName=...` — zurückziehen (nur Uploader)
- `GET /api/photos/list/[tripSlug]?viewerName=...` — sichtbare Fotos auflisten

### KV-Schema
```
photo:{id}          → SharedPhoto JSON (Hash)
photos:{tripSlug}   → Set aller Photo-IDs in dieser Reise
```

## Migrations-Hinweise

**WICHTIG**: Solange diese Env-Vars in Vercel NICHT gesetzt sind, geben
die API-Endpoints 503 zurück (mit hilfreichem Hinweis):

```
BLOB_READ_WRITE_TOKEN     → automatisch von Vercel Blob gesetzt
KV_REST_API_URL           → automatisch von Vercel KV gesetzt
KV_REST_API_TOKEN         → automatisch von Vercel KV gesetzt
KV_REST_API_READ_ONLY_TOKEN → automatisch von Vercel KV gesetzt (optional)
```

### Aktivierungs-Schritte (Vercel-Dashboard):
1. **Vercel Dashboard** → `Andreas60LondonApp` → **Storage** Tab
2. **„Connect Database"** → **„Blob"** auswählen → Region: Europe (Frankfurt) empfohlen
3. Erneut **„Connect Database"** → **„KV"** (oder „Upstash for Redis" über Marketplace) auswählen
4. Beide werden automatisch mit dem Projekt verbunden, Env-Vars erscheinen
5. **Redeploy** triggern (Deployments → letzter Deploy → Redeploy ohne Cache)

Ohne diesen Schritt: Backend antwortet mit 503, UI zeigt „Service nicht
verfügbar" — kein App-Crash, alles andere funktioniert weiter.

### Kosten:
- Vercel Blob: gratis bis 0.5 GB Storage + 5 GB Bandwidth/Monat
- Vercel KV: gratis bis 30k Commands + 256 MB Storage/Monat
- Bei 5 Reisende × 50 Fotos × ~1 MB → ca. 250 MB / weit innerhalb Free-Tier
