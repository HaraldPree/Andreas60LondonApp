# v1.20.0 — Service-Worker Schritt a (Static caching + Offline-Page)

**Datum**: 26.05.2026
**Status**: deployed
**Git Tag**: v1.20.0

## Anlass

Polarsteps-Benchmark schließen — die App muss auch funktionieren wenn
das Handy gerade kein Netz hat (Tube-Tunnel, Funkloch, Roaming-Pause).
Voraussetzung für „dein persönlicher Reisebegleiter": **Du musst auf
die App zählen können**, auch offline.

Schrittweise statt Big-Bang (siehe Diskussion mit Harald 26.05.):

| Iteration | Scope | Aufwand |
|---|---|---|
| **v1.20.0 (jetzt)** | Static assets + Offline-Fallback-Page | 1 Tag |
| v1.21.0 (geplant) | API-Caching (Wetter, Trip-Daten, Foto-Liste) | 1–2 Tage |
| v1.22.0 (geplant) | Foto-Thumbnail-Caching (volles Offline-Erinnerung) | 1–2 Tage |

Begründung schrittweise: Service-Worker-Bugs werden **permanent
gecacht** auf dem User-Gerät — wenn eine kaputte Version live geht,
sieht der User sie selbst nach 50 Reloads noch. Schrittweise =
Test-Stops zwischendurch = Rollback einfacher.

## Library-Wahl: next-pwa + Workbox

Gegen Vanilla-SW entschieden weil:
- **Battle-tested**: Workbox ist Google-Standard seit 2018
- **iOS Safari + Samsung Internet Edge-Cases** sind drin
- **Versionierte Caches** mit Auto-Purge alter Versionen
- **Update-Mechanik** (`skipWaiting`, `clientsClaim`) eingebaut
- **Bei Problemen**: weltweite StackOverflow-Hilfe

Vanilla wäre ~500 Zeilen Code statt ~50 Zeilen Config. Wir bauen
Travel-Memory, kein Service-Worker-Framework.

## Was ist neu

### `next.config.js` — next-pwa-Wrapper

```js
const withPWA = require("next-pwa")({
  dest: "public",
  disable: process.env.NODE_ENV === "development",
  register: true,
  skipWaiting: true,
  fallbacks: { document: "/offline" },
  runtimeCaching: [
    // Hero-Bilder + Icons       → CacheFirst (30 Tage)
    // OSM-Tiles (Leaflet)       → CacheFirst (14 Tage, max 200)
    // Fonts                     → CacheFirst (90 Tage)
    // /_next/static/ Bundles    → StaleWhileRevalidate (30 Tage)
    // HTML-Seiten               → NetworkFirst (5s Timeout, 14 Tage Cache-Fallback)
  ],
});
```

**Schritt-a-Caching-Strategien**:

| Assets | Strategie | Warum |
|---|---|---|
| Hero-Bilder, Icons, Avatare | CacheFirst | Ändern sich nie — Cache wins, kein Netz nötig |
| OSM-Map-Tiles | CacheFirst | Polarsteps-Parität: Karte funktioniert offline für besuchte Regions |
| Fonts (Playfair, DM Sans) | CacheFirst | Wenig Updates, sofort beim ersten Online-Besuch eingefroren |
| Next.js JS/CSS-Bundles | StaleWhileRevalidate | Schnell laden, Hintergrund-Update |
| HTML-Pages | NetworkFirst (5s) | Frisch wenn möglich, Cache-Fallback wenn Netz lahmt/weg |

### `src/app/offline/page.tsx` — Offline-Fallback (neu)

Hübsche Fallback-Seite mit:
- Markenname (aus Tenant)
- Cloud-Off-Icon + Erklärung „Du bist gerade offline"
- Hinweis: „Bereits besuchte Seiten funktionieren weiter, eigene
  Fotos liegen lokal"
- „Nochmal versuchen"-Button (window.reload)
- Link zur Reise-Übersicht
- Footer-Hinweis: Cache läuft beim nächsten Online-Besuch im
  Hintergrund

Wichtig: **Client-Component** (`"use client"`), weil onClick-Handler
für den Reload-Button gebraucht wird. Anders wirft Next.js
„Event handlers cannot be passed to Client Component props".

### `src/middleware.ts` — PIN-Gate-Whitelist erweitert

- `/offline` als public path (sonst Redirect-Schleife wenn der
  offline-User auf /login geleitet würde, was auch keine Verbindung
  hätte)
- `/sw.js`, `/workbox-`, `/fallback-` als public weil der SW selbst
  ja auch ohne PIN ausgeliefert werden muss

### `.gitignore` ergänzt

`public/sw.js`, `public/workbox-*.js`, `public/fallback-*.js` werden
bei jedem Build neu erzeugt und sollen nicht in git landen.

## Verhalten nach diesem Release

| Szenario | Vorher (v1.19.0) | Nach v1.20.0 |
|---|---|---|
| Online, App öffnen | Lädt vom Server | Lädt vom Server + cached für später |
| Offline, App schon mal besucht | Browser-Fehler-Page | App wird aus Cache geladen, vollständig sichtbar |
| Offline, neue Route nie besucht | Browser-Fehler-Page | `/offline`-Fallback mit Erklärung |
| Online, Update-Banner-Klick | Reload | Reload + neuer SW aktiviert sich |
| OSM-Karte offline | Leer / Fehler | Tile-Cache zeigt bereits gesehene Regionen |

## Wie testen

1. App online öffnen, einmal in jeden Tab klicken (Planen / Erleben /
   Erinnern / SOS / Info) — damit alles in den Cache geht
2. Flugmodus an (oder WLAN aus + Mobile-Netz aus)
3. App schließen + erneut öffnen
4. Durch alle Tabs scrollen → alles sollte funktionieren
5. Eine NIE besuchte Route aufrufen (z.B. `/agb` wenn man da noch
   nicht war) → sollte `/offline` zeigen
6. Wieder online → Cache-Refresh im Hintergrund

## Bekannte Limits

- **vulnerabilities in next-pwa-Deps**: `npm audit` zeigt 10
  Warnungen (1 moderate, 9 high) in transitiven Workbox-Paketen.
  Sind Build-Time-Tools — kein Runtime-Risiko für End-User.
  Nicht fixbar ohne Library-Wechsel.
- **`/manifest.webmanifest`** wird noch als trip-spezifisch gerendert
  ("Andreas 60. Geburtstag · London") — bei Multi-Tenant + Multi-
  Trip wird das dynamisch. Heute mit nur 1 Trip irrelevant.
