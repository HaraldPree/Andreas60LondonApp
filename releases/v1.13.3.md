# v1.13.3 — Header-z-index-Fix für Leaflet-Karte

**Datum**: 25.05.2026
**Status**: deployed
**Git Tag**: v1.13.3

## Anlass

Harald-Feedback nach v1.13.0-Release:

> Die interaktive Karte funktioniert — wenn ich hochscrolle läuft sie aber
> über dem Header, das ist ein Bug.

Reproduzierbar am Samsung A53 und im Chrome-DevTools-Mobile-View: beim
Hochscrollen im Karte-Tab scheint die Leaflet-Map durch den fixen Header
hindurch, statt dahinter wegzulaufen.

## Ursache

Leaflet vergibt seinen internen Map-Layern hohe `z-index`-Werte:

| Pane | z-index |
|---|---|
| `.leaflet-tile-pane` | 200 |
| `.leaflet-overlay-pane` | 400 |
| `.leaflet-marker-pane` | 600 |
| `.leaflet-popup-pane` | 700 |
| `.leaflet-control` | 1000 |

Der App-Header sitzt aber bei `z-40`. Beim Scroll kommt die Map-Tile
(z-200) über den Header (z-40) — sieht aus als würde die Karte den
Header überlagern.

## Lösung

Statt den Header auf `z-[1100]` hochzuziehen (und parallel **alle**
Modals/Sheets von z-[55]–z-[70] mitziehen zu müssen) reicht eine einzige
CSS-Regel:

```css
.leaflet-container {
  isolation: isolate;
}
```

`isolation: isolate` erzeugt für den Karten-Container einen neuen
**Stacking Context**. Leaflets interne z-Order stackt weiterhin korrekt
*innerhalb* der Karte (Marker über Tile, Popup über Marker), aber die
Karte als Ganzes hat im umgebenden Layout effektive z-index 0 und liegt
damit garantiert unter:

- dem `z-40`-Header,
- dem `z-40`-Bottom-Nav,
- allen `z-[55]`–`z-[70]`-Modals/Sheets (Companion, Reel, Action-Sheets).

Kein einziges anderes z-index musste angefasst werden.
