# v1.22.0 — Service-Worker Static-only Comeback

**Datum**: 26.05.2026
**Status**: deployed
**Git Tag**: v1.22.0

## Anlass

Nach der Open-Meteo-502-Erkenntnis (siehe v1.21.4): der „Wetter-Bug"
aus v1.20.0-Live-Test war NIE der Service-Worker, sondern eine externe
API-Outage. Der Static-Caching-Code aus v1.20.0 hat technisch
vermutlich funktioniert.

Konsequenz: SW-Initiative wieder einschalten, aber bewusst **nur den
Teil der schon funktioniert hat** — Static-Caching ohne Cross-Origin,
ohne eigene API. Cross-Origin-Caching (Open-Meteo, TfL) bleibt
deaktiviert weil es in v1.21.0 echt einen Workbox-Issue hatte (und
Wetter ohnehin jetzt via localStorage-Cache resilient ist).

## Was ist neu

### `next.config.js` — next-pwa wieder aktiv

`disable: process.env.NODE_ENV === "development"` (statt `true`).
Production-Build generiert wieder `public/sw.js` + workbox-Runtime.

**Caching-Strategien**:

| Pattern | Strategie | Max-Age | Begründung |
|---|---|---|---|
| `/images/**`, `/icons/**` (PNG/JPG/WebP/SVG) | CacheFirst | 30 Tage | Hero-Bilder + Avatare ändern sich kaum |
| `[abc].tile.openstreetmap.org/**` | CacheFirst | 14 Tage | OSM-Map-Tiles für besuchte Leaflet-Regions |
| Fonts (woff/woff2/ttf/otf/eot) | CacheFirst | 90 Tage | Fonts sind Build-Time-stabil |
| `/_next/static/**` (JS/CSS Bundles) | StaleWhileRevalidate | 30 Tage | Schnell laden + Hintergrund-Update bei neuem Build |
| HTML-Pages (App-Routes, nicht `/api`) | NetworkFirst (5s Timeout) | 14 Tage | Frisch wenn online, Cache wenn Netz weg/lahm |

**BEWUSST NICHT gecacht**:
- **Cross-Origin-APIs** (Open-Meteo, TfL) — war Bug-Quelle in v1.21.0
- **Eigene `/api/**`-Routes** — kommt frühestens v1.24.0 (wenn nötig)
- **`/api/version`** — sonst funktioniert Update-Detection nicht

**Plus**: `cleanupOutdatedCaches: true` — Workbox räumt alte Cache-
Versionen automatisch beim SW-Update.

### `ServiceWorkerCleanup.tsx` — mit Idempotenz-Flag

Cleanup-Hook bekommt `localStorage`-Flag
(`travelConcierge:sw-cleanup-done:v1`) und läuft jetzt **genau einmal
pro Browser**, danach nie wieder.

Vorher problematisch: bei jedem App-Open hätte der Hook auch den
NEUEN v1.22.0-SW direkt nach Install wieder gekillt. Jetzt:

- 1. Visit: Flag noch nicht gesetzt → Cleanup läuft (alte v1.20-v1.21.x SWs
  weg, Caches gelöscht, Tab reloaded). Flag gesetzt.
- 2.+ Visit: Flag gesetzt → Hook returns sofort → neuer SW darf leben

Bei künftigem grundlegendem SW-Reset (z.B. v1.30.0 mit komplett anderer
Caching-Strategie): `CLEANUP_DONE_KEY` auf `v2` incrementieren — alle
User durchlaufen Cleanup einmalig erneut. Sauberer Migrations-Pfad.

### `public/sw.js` Self-Destroying-Kill-Switch entfernt

Der manuelle Kill-Switch-SW aus v1.21.3 ist gelöscht — wird durch den
next-pwa-Output ersetzt. `.gitignore` ist zurück auf v1.20.0-Stand
(SW-Artefakte wieder ignoriert).

User mit altem v1.21.3-Kill-Switch im Browser:
- Kill-Switch hat sich bei Activate selbst dereinstaliert + Caches gelöscht
- Browser ist SW-frei
- Beim v1.22.0-Visit: neuer Workbox-SW wird normal installiert + aktiv

User mit noch-älterem v1.20.0/v1.21.0-SW (haben nie v1.21.2-1.21.4
gesehen, unwahrscheinlich aber möglich):
- Browser holt das neue `/sw.js` (von next-pwa generiert)
- Hash unterscheidet sich → neuer SW wird installiert
- `cleanupOutdatedCaches: true` räumt v1.21.x-Caches auf
- ServiceWorkerCleanup-Hook setzt Flag + räumt eventuelle Reste

## Was du sehen wirst

| Szenario | Verhalten |
|---|---|
| Online, App öffnen | Lädt vom Server + cached für später |
| Offline, App vorher mal besucht | Layout + Tabs + Karten-Tiles aus Cache → komplett bedienbar |
| Offline, NIE besuchte Route | `/offline`-Fallback mit Hinweis |
| OSM-Karte für besuchte Region | offline sichtbar |
| Wetter offline | Aus `useWeather`-localStorage-Cache (v1.21.4), max 24h alt, mit klarem Status-Footer |
| Wetter online + API live | normal frisch |
| Wetter online + API tot | localStorage-Cache + Status-Footer „Wetter-Server hat Probleme" |
| AI-Companion-Chat | online-pflichtig (LLM-Cost) |

## Was bewusst NICHT in dieser Iteration

- **Cross-Origin-API-Caching** (Open-Meteo, TfL via SW) — kommt
  vermutlich nie zurück; bessere Architektur ist `/api/weather`-
  Server-Proxy (= Same-Origin)
- **Eigene API-Routes im SW-Cache** — heute funktional unproblematisch
  ohne, kommt nur wenn echter Bedarf
- **Background-Sync** für Foto-Uploads — Phase 2

## Update-Flow

User mit v1.21.4 → v1.22.0:
1. Vercel-Deploy aktiv
2. Browser checkt `/api/version` (existing) → sieht neuen Build-SHA
3. UpdateBanner → User klickt → Reload
4. Browser checkt `/sw.js` parallel → findet neuen Workbox-SW
5. Install + skipWaiting → SW aktiviert
6. `cleanupOutdatedCaches` räumt alte Workbox-Caches (falls da)
7. `ServiceWorkerCleanup`-Hook: Flag schon gesetzt von v1.21.2-Cleanup-
   Lauf → Hook returns sofort, neuer SW läuft
