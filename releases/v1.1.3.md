# v1.1.3 — Foto-Sharing Blob-only (KV nicht mehr nötig)

**Datum**: 21.05.2026
**Status**: deployed
**Git Tag**: v1.1.3
**Bump-Grund**: PATCH — interne Architektur-Vereinfachung, kein User-Feature-Change

## Was ist neu

Refactor von **Blob + KV** auf **Blob-only**. Foto-Sharing funktioniert
jetzt mit NUR Vercel Blob — eine zweite Storage-Datenbank (KV / Upstash
Redis) ist nicht mehr nötig.

### Architektur-Änderung

**Vorher (v1.0.1 – v1.1.2)**:
```
Blob          → speichert Foto-Bytes (Voll + Thumb)
KV (Redis)    → speichert Metadaten (Hash photo:{id})
                + Index (Set photos:{tripSlug})
```

**Jetzt (v1.1.3+)**:
```
Blob          → speichert SOWOHL Fotos als auch Metadaten
                
                trips/{slug}/{id}-full.jpg
                trips/{slug}/{id}-thumb.jpg
                trips/{slug}/manifest-{ts}.json   ← die Metadaten
```

Pro Reise gibt es **eine Manifest-JSON-Datei** die als „Index" dient.
Beim Upload wird sie gelesen, der neue Eintrag angehängt, und wieder
zurückgeschrieben.

### Trade-offs

| Aspekt | Bisher (KV) | Jetzt (Blob-only) |
|---|---|---|
| Schreib-Latency | ~50ms | ~200ms |
| Lese-Latency | ~30ms | ~100ms |
| Race-Condition | atomar sicher | klein bei gleichzeitigen Uploads |
| Setup-Aufwand | 2 Services | 1 Service |
| Kosten | beide free-tier | nur Blob (free-tier) |
| Skalierung | beliebig | optimal bis ~500 Fotos pro Reise |

Für 5-Personen-Reise mit ~100 Fotos: keine spürbaren Nachteile.

### Geänderte Dateien

- `src/lib/sharedPhotoStore.ts` — kompletter Rewrite mit Manifest-Pattern
  (`readManifest`, `writeManifest` helper). `getSharedPhoto` und
  `updateSharedPhotoVisibility` brauchen jetzt zusätzlich `tripSlug` als
  Argument weil das Manifest pro Reise gespeichert ist.
- `src/app/api/photos/share/route.ts` — PATCH und DELETE Routen lesen
  `tripSlug` aus den Query-Params + reichen es an die Store-Funktionen
  durch.
- `src/hooks/useSharedPhotos.ts` — `changeVisibility` und `withdraw`
  übergeben `tripSlug` an die API.
- `src/app/api/version/route.ts` — `photoSharingReady` braucht jetzt
  nur noch `blobConfigured`. `kvConfigured` bleibt zur Information
  sichtbar (falls KV doch parallel da ist).

### Was bleibt gleich

- API-Endpunkte unverändert (POST/PATCH/DELETE/GET)
- Sichtbarkeits-Logik (`canViewSharedPhoto`) identisch
- Frontend-Komponenten unverändert (SharedGallery, PhotoShareSection,
  VisibilitySelector, SharingConsentModal)
- DSGVO-Aspekte: sofortige Löschung bei Widerruf, Privacy by Default,
  Einwilligung pro Foto — alle wie gehabt

## Was funktioniert NICHT

Kleines Race-Condition-Risiko: Wenn 2 Personen exakt gleichzeitig (im
ms-Bereich) ein Foto uploaden, kann der erste Eintrag verloren gehen
weil der zweite das vorherige Manifest überschreibt. Für 5-Personen-
Gruppe extrem unwahrscheinlich. Bei realistischen Verkaufs-Apps mit
mehr Concurrent-Users müsste dies via optimistic-concurrency oder
"locking by lease" gelöst werden (z.B. via Vercel Blob `If-None-Match`-
Header oder doch KV als Lock-Service).

## Migrations-Hinweise

### Storage-Setup
Nur **Vercel Blob** muss in Vercel Dashboard verbunden sein:
1. Storage → Connect Database → Blob → Frankfurt → Create + Connect
2. Redeploy (ohne Cache-Häkchen)

KV / Upstash KANN parallel verbunden sein (schadet nicht), wird aber
nicht mehr genutzt. `@vercel/kv` ist als npm-Dependency noch da,
kann später entfernt werden.

### Verifizieren
`/api/version` zeigt:
```json
{
  "blobConfigured": true,
  "kvConfigured": false,        // wird ignoriert
  "photoSharingReady": true     // ← Ziel
}
```

## User-Kommunikation

Keine direkte User-Notification — Funktionalität exakt gleich. Sobald
deployed + Blob aktiviert, sehen alle Reisenden die "Gemeinsame
Galerie" als nutzbar (statt der "wird eingerichtet"-Box).
