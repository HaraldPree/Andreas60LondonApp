# v1.5.0 — Mehrere Fotos auf einmal teilen (iOS-Photos Selection-Mode)

**Datum**: 19.05.2026
**Status**: deployed
**Git Tag**: v1.5.0
**Bump-Grund**: MINOR — neues User-Feature
**Vorgeschlagen von**: Mitreisende (live, 19.05.2026)

## Was ist neu

Foto-Sharing kann jetzt **mehrere Fotos gleichzeitig** teilen — bisher
ging nur eins nach dem anderen. Konkrete Kritik aus der Reisegruppe:
*„bei bilder für alle teilen immer nur 1 bild geht es sollten mehrere
gleichzeitig funktionieren das wäre viel besser"*

### Wie's funktioniert (Apple-Photos-Pattern)

1. Im **Fotos-Tab** rechts oben Button „✓ Auswählen" antippen (vormals
   „Bearbeiten")
2. **Sticky-Action-Bar** erscheint oben unter der Sektions-Überschrift
   mit Zähler und Aktionen
3. Auf Fotos tippen → Foto bekommt **Checkmark-Overlay**, Ring um die
   Kachel, leichtes Brightness-Dimming
4. Wenn ≥1 ausgewählt: Buttons in der Bar werden aktiv
   - **„Teilen"** → öffnet das `BulkShareSheet`
   - **„Löschen"** → Confirm + Batch-Delete via `removeMany`
5. „Fertig"-Button rechts oben beendet den Auswahl-Modus

„Verstanden"-Button.

### Slim für Backend, smart pro Foto

Das Sheet erkennt pro Foto ob es **schon online** ist und ruft dann nur
`changeVisibility()` statt erneutem Upload — sparsam mit Bandbreite +
Blob-Storage. Logik:

```typescript
const sharedIds = new Set(alreadyShared.map((p) => p.id));
for (const photo of selectedPhotos) {
  if (sharedIds.has(photo.id)) {
    await changeVisibility(photo.id, newVisibility);  // schnell
  } else {
    await shareOne({ ...fullUploadArgs });            // langsamer
  }
}
```

### DSGVO / Consent

Genau wie beim Single-Photo-Sharing: beim ersten Bulk-Teilen wird die
AGB+Datenschutz-Einwilligung via `SharingConsentModal` einmal abgefragt.
Danach läuft jeder weitere Bulk-Upload ohne Modal.

### Selection-Mode ersetzt Edit-Mode

Der frühere **X-pro-Foto Quick-Delete-Modus** ist weg. Stattdessen
**eine Mode für beides**: Bulk-Teilen UND Bulk-Löschen. Konsistent
mit iOS Photos.

„Alle Fotos dieser Reise löschen" als nukleare Option bleibt am Ende
der Liste (nur im Selection-Mode sichtbar).

## Was funktioniert NICHT (in dieser Version)

- **Sequenzieller Upload** ist langsamer als parallel — bei 20 Fotos
  könnte das 30+ Sek dauern. Bewusste Entscheidung: schont das Netz auf
  Mobil-Verbindungen + reduziert Rate-Limit-Risiken bei Vercel Blob.
  Parallel mit `Promise.allSettled` wäre eine künftige Optimierung.
- **Keine Cancel-Möglichkeit während Upload läuft**. Wenn der User auf
  den Backdrop tippt, wird das Sheet beim Hochladen NICHT geschlossen
  (verhindert Datenverlust). Wer wirklich abbrechen muss: App schließen
  und neu laden, der Server hat dann nur die schon hochgeladenen Fotos.
- **Geteilte Fotos im SharedGallery-Refresh**: Der Hook `useSharedPhotos`
  hat seinen eigenen Cache, der nach jedem `shareOne` / `changeVisibility`
  refresht. Bei 5 Photos = 5 Refreshes hintereinander. Funktional OK,
  aber etwas Overhead. Optimierung später möglich.
- **Selection-Bar ist sticky in der Tab, nicht im Viewport**. Bei sehr
  langer Tab-Scrolling-Position bleibt die Bar weiterhin sichtbar weil
  sticky-top — das passt für mobile, ist aber kein Pattern wie ein
  Bottom-Sheet.
