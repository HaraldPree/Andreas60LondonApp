# v1.14.3 — Cluster-Split bei GPS-losen Fotos + Programm-Item-Fallback

**Datum**: 25.05.2026
**Status**: deployed
**Git Tag**: v1.14.3

## Anlass

Harald-Feedback nach v1.14.2-Live-Test:

> Bei Erlebt steht immer noch „Stopp ohne GPS" überall, und teilweise
> bis zu 111 Fotos je Stop.

Beide Symptome haben **dieselbe Ursache**: Fotos ohne EXIF-GPS.

- v1.14.2's Cluster-Lockerung (60 min + 400 m) prüfte Distanz nur
  wenn beide Fotos GPS hatten. Wenn keines GPS hatte, war `distOk=true`
  per Default → alles innerhalb von 60 min landete im selben Cluster.
- Über einen ganzen Reisetag mit ~111 Fotos im 60-min-Raster wuchs
  damit ein einziger Mega-Cluster — entsprechend stand „111 Fotos"
  beim einzigen Stop.
- Reverse-Geocoding (v1.14.2) konnte nicht helfen weil kein GPS = keine
  Koords zum Geocoden.

## Was geändert wurde

### 1. Strikteres Clustering ohne GPS

`src/lib/tripReconstruction.ts`:

Neue Konstante `CLUSTER_TIME_GAP_NO_GPS_MIN = 25` (Min). Wird angewendet
wenn min. eines der zwei verglichenen Fotos kein GPS hat:

```ts
const bothHaveGps = !!photo.coordinates && !!lastPhoto.coordinates;
const effectiveTimeGapMin = bothHaveGps
  ? CLUSTER_TIME_GAP_MIN          // 60 Min (Distance-Check als Backup)
  : CLUSTER_TIME_GAP_NO_GPS_MIN;  // 25 Min (kein Backup → enger)
```

Wirkung: ein typischer Sightseeing-Tag (Frühstück, Cedric Grolet,
Hyde Park, V&A, Notting Hill, Dinner) zerfällt jetzt auch ohne GPS in
~5–7 vernünftige Cluster — nicht mehr in einen einzigen Bulk.

### 2. Programm-Item-Match als zweite Fallback-Schicht

Drei-Stufen-Match-Strategie pro Cluster (vorher zwei):

| Stufe | Quelle | Wann aktiv |
|---|---|---|
| 1 | **GPS → Place-Library** (kuratiert) | Cluster hat GPS-Center + Place ≤ 300 m |
| 2 | **GPS → Nominatim** (Stadtteil-Label) | Cluster hat GPS-Center, kein Place-Match, Nominatim findet was (v1.14.2) |
| 3 (NEU) | **Zeit → `trip.days[d].items`** | Programm-Item-Label am nächsten zur Cluster-Startzeit (≤ ±90 min) |

Helper `matchProgramItemByTime()`:
- Parsed Programm-Item-Zeiten: numerisch (`"13:00"`) UND Tagesabschnitt
  (`"Vormittag"` → 10:00, `"Nachmittag"` → 15:00, `"Abend"` → 19:00 etc.)
- Findet das zeitlich nächstgelegene Item
- Match nur akzeptiert wenn ≤ `PROGRAM_ITEM_MATCH_WINDOW_MIN` (= 90 min)
- Übernimmt `label`, `icon`, `placeId` (für Klick-Verlinkung)

**Konkrete Wirkung**: ein Cluster um 13:00 am Dienstag bekommt jetzt
„🥐 Cedric Grolet @ The Berkeley (goûtea)" als Name + Icon — statt
„Stopp ohne GPS". Genau das was Harald sehen wollte.

### 3. Hook `useReconstructedTrip.ts`

Reicht `trip.days[dayIdx]?.items` als dritter Param an
`reconstructDay()` durch. Eine Zeile Aufruf, Hook-Cache (useMemo) bleibt
unverändert.

## Was funktioniert NICHT (geplant für später)

- **Falsche Item-Matches** bei untypischen Foto-Zeitpunkten: wenn jemand
  um 15:00 plötzlich ein Foto vom „Pub Dinner Notting Hill" (19:30)
  macht, matched das auf den nächsten Item — eventuell „Hyde Park" um
  14:45. Da das Item-Window 90 Min ist, wird der Pub-Eintrag nicht
  fälschlich getroffen. Bei Sub-30-Min-Items kann es trotzdem zu
  Vorgriffen kommen. Wenn das zu oft passiert: Window verkleinern oder
  pro Item Toleranz aus `duration` ableiten.
- **Item-Match liefert kein GPS-Distanz-Feedback** — das Distance-
  Label „· 80 m" wird nur bei Place-Match angezeigt. Item-Match ist
  zeitbasiert, keine Distanz möglich.
- **Mega-Foto-Cluster (111 Fotos)** bleiben sichtbar wenn legitim
  (z.B. 2 Stunden in einem Museum mit viel knipsen). Der „+N"-Badge
  fängt die Optik aber ab — nur 3 Thumbnails sichtbar, Rest als
  Zahl.
