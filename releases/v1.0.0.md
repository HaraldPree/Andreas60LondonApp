## Was funktioniert NICHT (bekannte Limits)

- **Foto-Sharing zwischen Reisenden**: nicht implementiert — jedes Handy hat eigene Foto-Sammlung. Workaround: manueller ZIP/PDF-Export + Versand via WhatsApp/Drive. Foto-Sharing-Backend ist Phase-1 nach der Reise.
- **Programm-Änderungen während Reise**: müssen via Code-Push gemacht werden (Harald sagt Bescheid, Claude pusht). Editor-UI kommt in Phase-1.
- **Gruppen-Chat**: extern via WhatsApp. Kein eigener Chat in der App.
- **Personal-Overlay für Programm**: bewusst NICHT gebaut (Gruppen-Konsens: zentralisiert ist besser).
- **HEIC-Fotos**: werden mit freundlicher Fehlermeldung abgelehnt (kein Browser-Decoder).
- **Offline-Modus**: teilweise vorhanden (Programm-Texte, Phrasebook, Reservierungen) aber keine echte Service-Worker-Strategie.
- **Multi-Trip**: nur eine Reise pro Deployment, die slug ist hart in `src/data/trips/`.

## Migrations-Hinweise

- **PIN**: muss in Vercel als `APP_PIN` Env-Var gesetzt sein (Production scope!)
- **APP_ANTHROPIC_KEY**: für KI-Companion erforderlich
- **AVIATIONSTACK_API_KEY**: optional, nur für Live-Flugstatus
- **NEXT_PUBLIC_HARALD_WHATSAPP**: optional, sonst Default aus Code
- Nach Env-Var-Änderung in Vercel: **Redeploy nötig** (Env-Vars werden nicht im laufenden Deployment aktualisiert)
