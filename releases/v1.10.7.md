# v1.10.7 — Save-Pfade umgebaut: was Samsung kann zuerst

**Datum**: 23.05.2026
**Status**: deployed
**Git Tag**: v1.10.7

## Anlass

Harald-Test mit Samsung A53 nach v1.10.6 + Anhang (document(11).pdf):

> „pdf Speichern teilen sheet funktioniert nicht
> pdf in teilen ... 2. punkt hat funktioniert
> 3. punkt funktioniert auch nicht"

Klare Hierarchie auf Samsung Internet:
| Button | Status |
|---|---|
| 🟦 „Teilen-Sheet" (Share-API) | ❌ stillschweigend nichts |
| 🟨 „In neuem Tab" (window.open) | ✅ funktioniert |
| 🩶 „Direct-Download" (anchor) | ❌ about:blank |

## Fix — UI nach Funktionalität sortiert

Aus „3 Buttons probieren bis einer geht" wird:

**PRIMARY** (navy, groß, ganz oben):
- „PDF öffnen + speichern" → `window.open(blob:url)` → PDF rendert inline →
  Browser-Menü → „Seite speichern"

**SECONDARY** (gold, kleiner):
- „Teilen / Senden (iOS Safari, WhatsApp)" → `navigator.share()` → Share-Sheet
- Bewusst kleiner gestylt: funktioniert auf iOS Safari sehr gut, auf Samsung
  Internet buggy

**ENTFERNT**:
- „Direct-Download (Desktop / Chrome)" — funktionierte auf Samsung nicht,
  auf Desktop kommt User auch über die anderen beiden Wege zum Ziel

## Warum diese Reihenfolge

Praktische Funktionalität nach Browser:

| Browser | Open-in-Tab | Share-Sheet |
|---|---|---|
| Samsung Internet | ✅ | ❌ Bug |
| iOS Safari | ✅ | ✅ optimal |
| Chrome Android | ✅ | ✅ |
| Chrome Desktop | ✅ (kein download, nur view) | ✅ (OS-Sheet) |
| Edge Desktop | ✅ | ✅ |

Open-in-Tab hat 100% Funktionalität, Share-Sheet 75%. Daher Open-in-Tab
als Primary.

Trade-off: auf iOS Safari ist Share-Sheet eleganter (direkt „In Dateien
speichern" wählen statt Umweg über Browser-Menü). Aber als sekundärer
Button immer noch verfügbar.
