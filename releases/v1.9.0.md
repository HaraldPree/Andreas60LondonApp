# v1.9.0 — Events-Datenklasse + London-Event-Library

**Datum**: 23.05.2026
**Status**: deployed
**Git Tag**: v1.9.0

## Anlass

Während der London-Reise (18.–22. Mai 2026) lief direkt die **Chelsea Flower
Show** vom 19.–23. Mai — also genau in unserem Reisezeitraum. Die App hat
darauf **nicht hingewiesen**. Harald-Feedback nach Heimkehr:

> „Eines hat mich gestört — es gab die Woche Chelsea Bloom und du hast uns
> darauf nicht hingewiesen. Wie könnte man so ein tolles Event vor Ort finden,
> wenn wir gar nicht wissen dass es das gibt?"

Die Place-Library hat nur **permanente Sehenswürdigkeiten** abgebildet —
zeitlich gebundene Events (Festivals, Sport, Konzerte, Sonderausstellungen)
fielen komplett durch.

## Was ist neu

### 1. Event-Datenklasse (`src/types/event.ts`)

Neue erste-Klasse-Datenstruktur für zeitlich gebundene Aktivitäten:

```ts
interface Event {
  id, name, category, icon?
  startDate, endDate              // ISO YYYY-MM-DD
  location, coordinates?
  description, visitorTips?
  bookingUrl?, cost?, bookingRequired?
  recurring?                       // annual-fixed-week, one-off, etc.
  lastVerified, source             // Anti-Halluzinations-Pflichtfelder
  city?, tags?
}
```

**Drei Daten-Klassen** werden unterschieden:
1. **Annual-Recurring**: Chelsea Flower Show, Wimbledon, Notting Hill Carnival, Trooping the Colour
2. **Saisonal**: Winter Wonderland, Sakura, Bonfire Night
3. **One-Off**: Konzerte, Pop-Ups, Premieren

**10 Kategorien**: festival, exhibition, sport, music, culture, seasonal,
market, fireworks, pride, other.

**Helper-Funktionen**:
- `eventOverlapsTrip(event, tripStart, tripEnd)` — matched Event gegen Reise-Zeitraum
- `eventActiveOnDate(event, date)` — matched Event gegen Einzeltag

### 3. Trip-Schema-Erweiterung (`src/types/trip.ts`)

```ts
interface Trip {
  ...
  events?: Event[];   // optional, fällt auf [] zurück
}
```

Backwards-kompatibel — bestehende Trip-Files brauchen das Feld nicht.

### 4. Event-Banner Komponente (`src/components/trip/EventBanner.tsx`)

Zeigt im **Programm-Tab** alle Events die mit dem Reisezeitraum überlappen:

- Filter: `eventOverlapsTrip()` gegen `tripStart`/`tripEnd` aus `days[].isoDate`
- Sortierung: nach `startDate` aufsteigend
- Null-Fallback: wenn keine matchen → Banner unsichtbar
- Expandable-Rows: Tap aufs Event → Details, Visitor-Tipps, Booking-Link
- Overlap-Anzeige: „· 3 Tage eurer Reise" (zählt Tage-Schnittmenge inklusive)
- Quelle + lastVerified im Footer jedes Events (Anti-Halluzinations-Disziplin)

Visuell: gold/cream Gradient mit Sparkles-Icon, sitzt direkt unter TripHero.

### 5. Integration in ProgrammTab

`<EventBanner trip={trip} />` rendert zwischen `<TripHero>` und
Goodbye-Reel-Banner. Null-Render wenn keine Events matchen.

## Was funktioniert NICHT (bekannte Limits)

- **Heute nur London-Events** — andere Reiseziele brauchen eigene
  `[stadt]-events.ts` Datei
- **Manuelle Pflege** — die Events wurden händisch recherchiert und in TS
  hardgecodet. Annual-Events müssen jedes Jahr aktualisiert werden
  (lastVerified-Pflicht)
- **Keine Auto-Recherche** — das kommt in v1.10.0 (AI-Pre-Trip-Research)
- **Kein User-Vorschlag** — User können (noch) keine Events einreichen
