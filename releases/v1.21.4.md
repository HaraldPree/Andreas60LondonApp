# v1.21.4 — Wetter-Cache + bessere Error-Message bei API-Ausfall

**Datum**: 26.05.2026
**Status**: deployed
**Git Tag**: v1.21.4

## Anlass — die Lehre aus dem Open-Meteo-Vorfall

User-Bericht v1.21.3: „Wetter konnte nicht geladen werden, auch nach
Cache-Cleanup." → Diagnose via `curl`:

```
api.open-meteo.com  →  HTTP 502 Bad Gateway (3 von 3 Versuchen)
open-meteo.com      →  HTTP 200 OK
```

**Open-Meteo API selbst ist down**, nicht unser Service-Worker, nicht
unser Code. Wir haben einen halben Tag SW-Bugs gejagt (v1.20.0 →
v1.21.0 → v1.21.1 → v1.21.2 → v1.21.3) für einen externen API-Ausfall.

### Lessons Learned

1. **IMMER zuerst externe Abhängigkeit prüfen** bevor eigener Code
   debuggt wird. `curl api.open-meteo.com` hätte das in 2 Sekunden geklärt.
2. **Defensiv cachen** — App muss bei API-Ausfällen gracefully degradieren.
3. **Sichtbare Fehler-Status** — User soll sofort sehen ob „mein Bug"
   oder „externer Aussetzer".

## Was geändert

### `src/lib/weather.ts`

**localStorage-Cache** für Wetter-Daten:

```ts
export function loadCachedWeather(lat, lng): { data, cachedAt, ageMs } | null
export function saveCachedWeather(lat, lng, data): void
export function formatRelativeAge(ms): string  // „vor 5 Min", „vor 2 Std"
```

- Cache-Key: `weather:cache:<lat>:<lng>` mit 4 Nachkommastellen
- Max-Age: **24 Stunden**, danach gilt der Cache als veraltet
- Storage: `localStorage`, kein Server-Backend nötig

**Bessere Error-Messages** im fetchWeather:

| HTTP-Status | Alte Message | Neue Message |
|---|---|---|
| 502/503/504 | "Wetter konnte nicht geladen werden (502)" | "Wetter-Server hat gerade Probleme (HTTP 502 Bad Gateway)" |
| 429 | "Wetter konnte nicht geladen werden (429)" | "Zu viele Wetter-Anfragen — kurz warten + erneut versuchen" |
| sonst | "Wetter konnte nicht geladen werden (XXX)" | "Wetter konnte nicht geladen werden (HTTP XXX)" |

### `src/hooks/useWeather.ts` (komplett umgebaut)

Neuer State + Verhalten:

```ts
interface UseWeatherResult {
  data: WeatherData | null;
  loading: boolean;
  error: string | null;
  cachedAt: number | null;   // Zeitstempel der aktuellen Daten
  fromCache: boolean;        // true = aus localStorage, false = live
  refresh: () => Promise<void>;
}
```

Initial-Pfad:
1. Beim Mount: synchron localStorage-Cache lesen (falls jünger als 24h)
2. State direkt mit Cache-Daten initialisieren → User sieht sofort Wetter
3. Im Hintergrund: Live-Fetch starten
4. **Erfolg** → Live-Daten ersetzen Cache, `fromCache=false`
5. **Fehler** → Cache bleibt, `error` wird gesetzt, `fromCache=true`

### `src/components/weather/WeatherWidget.tsx`

Drei klare Zustände:

**Zustand 1 — Live-Daten ohne Probleme**
Standard-Widget, kein Status-Hinweis.

**Zustand 2 — Cache-Daten oder Fehler trotz Cache**
Widget zeigt Daten + Status-Footer:
- Wenn nur Cache (kein Live): kleines „Lokal gespeichert · vor X Min"
- Wenn Fehler + Cache: rot „Wetter-Server hat gerade Probleme … Zeige
  gespeicherten Stand von vor X Min."

**Zustand 3 — Fehler ohne jeden Cache (1. App-Visit + API tot)**
Kompakte Error-Karte mit:
- Header „Wetter aktuell nicht erreichbar"
- konkrete API-Fehler-Message (z.B. „Wetter-Server hat gerade Probleme (HTTP 502 Bad Gateway)")
- Hinweis „Externer Wetter-Anbieter (Open-Meteo) — wir versuchen automatisch alle 30 Min"
- Refresh-Button

## Wirkung im aktuellen Open-Meteo-Ausfall

| Vorher (v1.21.3) | Jetzt (v1.21.4) |
|---|---|
| „Wetter konnte nicht geladen werden" — keine Hinweise woher | „Wetter-Server hat gerade Probleme (HTTP 502 Bad Gateway)" |
| Bei nächstem App-Open weiterhin leer | Bei nächstem App-Open: letzter bekannter Stand sichtbar (wenn schon mal geladen) |
| User denkt „App kaputt" | User sieht „API tot, App OK" |

## Test

1. **Wenn Open-Meteo gerade tot ist** + du noch nie Wetter geladen hattest:
   → siehst kompakte Error-Karte mit „Wetter-Server hat gerade Probleme (HTTP 502)"
2. **Wenn Open-Meteo gerade tot ist** + du hast Wetter mal geladen
   (vor dem Ausfall):
   → siehst normales Widget mit Wetter + roten Status „Zeige gespeicherten Stand von vor X Min"
3. **Wenn Open-Meteo wieder online**:
   → normales Widget ohne Status-Footer

## Service-Worker — Re-Evaluation

Da der „Bug" gar nicht der SW war, sondern Open-Meteo, ist die ganze
SW-Initiative neu zu bewerten:

- **v1.20.0 Static-Caching** hat vermutlich **technisch funktioniert** — wir
  haben es nur falsch interpretiert weil zur selben Zeit Open-Meteo
  Probleme hatte
- **v1.21.0 API-Caching** (Open-Meteo + TfL via Workbox) hatte einen
  Cross-Origin-Issue — aber vermutlich ein anderer Bug, nicht der den
  User gesehen hat (er hat Open-Meteo gesehen)

Vorschlag: **v1.22.0** = SW Static-only zurück (jetzt mit dem Wissen
dass es vermutlich nicht das eigentliche Problem war). Aber erstmal
Open-Meteo abwarten + Cache-Verhalten in v1.21.4 verifizieren.
