# v1.2.4 — PWA-Retour-UX (Sentinel-History für Direkt-Open)

**Datum**: 19.05.2026
**Status**: deployed
**Git Tag**: v1.2.4
**Bump-Grund**: PATCH — kleiner UX-Fix für PWA-Direct-Open

## Was wurde geändert

Martin (Firefox Android, App via Home-Screen-Shortcut) hat ein Detail
entdeckt: wenn die App **direkt auf der Reise-Seite** startet (laut
`manifest.ts` ist `start_url: "/london-2026"`), hat die Browser-History
keinen vorherigen Eintrag. Die Retour-Geste findet nichts zum Popen —
Firefox Android zeigt grau, iOS Safari schließt die PWA, Chrome zeigt
den PWA-Switcher.

Wer dagegen via "/" → "/london-2026" reinkommt hat eine 2-Eintrag-
History und Retour funktioniert natürlich (zur Übersicht).

### Lösung

Neuer Hook `src/hooks/useTripPageBackToHome.ts`:
- Beim Mount der Reise-Seite prüfen: `window.history.length === 1`?
- Wenn ja (= PWA-Direct-Open ohne vorherige Navigation): einen Sentinel-
  State `{ rcmkTripBackToHome: true }` pushen
- Bei popstate (Retour-Geste): zur Übersicht `/` navigieren
- Wenn ein höheres Overlay vorher mit `useDismissOnBack` ein eigenes
  Sentinel gepusht hatte, fängt unseres den `popstate` NICHT ab — das
  Modal schließt sauber, und erst ein zweiter Back kommt zu uns

### Verschachtelungs-Test (PWA-Direct + Modal)

```
History: [null, tripSentinel, modalSentinel]   active = modalSentinel
   │
   └─ Back-Geste #1:  → active = tripSentinel
                       Modal-Handler: nextStateId=undef → schließt Modal ✓
                       Trip-Handler: flag=true → bleibt ruhig ✓
                       Modal verschwindet, User auf Trip-Seite
   │
   └─ Back-Geste #2:  → active = null
                       Trip-Handler: flag=undef → router.replace("/") ✓
                       App zeigt Übersicht
```

### Eingebaut in

`src/components/TripPageClient.tsx`: ein einzeiliger `useTripPageBackToHome()`-
Aufruf direkt nach dem Variant-Hook.

## Was funktioniert NICHT

- **Nicht aktiv** wenn der User über "/" reinkommt (history.length > 1).
  Das ist Absicht — dort funktioniert Retour von selbst.
- **router.replace** statt `router.push`: ersetzt den aktuellen
  History-Eintrag mit "/" statt einen neuen zu pushen. Falls der User
  von / zurück will gibt es keine Phantom-Einträge.
- **Bei Verschachtelung** mit `useDismissOnBack`: funktioniert. Wenn
  beim Testen doch Edge-Cases auftauchen (z.B. dreifach verschachtelte
  Overlays), passen wir an.
