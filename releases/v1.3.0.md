## Was ist neu

Reisende können **pro Item im Tagesprogramm** drei Aktionen pflegen,
ohne Claude zu pingen oder einen Code-Change abzuwarten:

1. **„✓ Erledigt"** — was wir tatsächlich gemacht haben
2. **„⊘ Ausgelassen"** — was wir bewusst nicht gemacht haben
3. **„✏️ Notiz"** — freie Notiz pro Item (z.B. „war super", „abgebrochen
   weil zu voll")

Speicherung pro Gerät in localStorage, kein Backend-Sync nötig. Variant-
aware (eigene IDs für Original vs. Alternative).

### Wie's aussieht

Jedes Item bekommt zwei kleine Buttons am rechten Rand:

```
09:00                                      ○ ⋯
Frühstück im Apartment
Apartment hat super Kaffee

10:00                                      ✓ ⋯       ← erledigt
T̶r̶a̶f̶a̶l̶g̶a̶r̶ ̶S̶q̶u̶a̶r̶e̶                          (durchgestrichen, opacity 70%)

11:30                                      ⊘ ⋯       ← ausgelassen
B̶u̶c̶k̶i̶n̶g̶h̶a̶m̶ ̶P̶a̶l̶a̶c̶e̶                        (grau, opacity 60%)
  ✍️ "wegen Wachablöse-Stress übersprungen"  ← eigene Notiz unterhalb
```

- **○ ✓ ⊘ Circle-Button**: Tap = Toggle „erledigt" (idle ↔ done)
- **⋯ Menu-Button**: Tap = Bottom-Sheet mit allen Optionen (erledigt,
  ausgelassen, Notiz, Markierung entfernen)

### Stats-Badge im DayCard-Header

Sobald mindestens 1 Item markiert ist, erscheint unter dem Tag-Titel
ein Status-Streifen:

```
✓ 3   ⊘ 1   / 7 Punkte
```

Plus „Tag zurücksetzen"-Link unten im aufgeklappten Day-Card.

### localStorage-Schema

Key: `rcmk:items:{tripSlug}`
Value:
```json
{
  "v=original:d=1:i=3": {
    "mark": "done",
    "note": "war super!",
    "updatedAt": 1716120000000
  },
  "v=alternative:d=3:i=2": {
    "mark": "skipped",
    "updatedAt": 1716121234567
  }
}
```

Variant-Prefix verhindert Kollisionen zwischen unterschiedlichen Programm-
Versionen (Donnerstag Original ≠ Donnerstag Sonnentag-Bündelung).

### Item-ID Schema

Items haben keine intrinsischen IDs in den Trip-Daten. IDs werden zur
Laufzeit aus `(variant, dayIndex, itemIndex)` konstruiert. Wenn Items
in den Daten umsortiert werden, kann State auf falsche Items zeigen —
akzeptable Schwäche für v1.3.0. Bei Bedarf könnten künftige Trip-Daten
explizite `id`-Felder pro Item bekommen.

## Was funktioniert NICHT (in dieser Version)

- **Kein Sync zwischen Reisenden** — Phase 3 aus dem Todo-Doc.
  Andrea sieht andere Marks als Harald. Bewusste Entscheidung: jede:r
  hat ihre persönliche Sicht (analog zur Variant-Wahl seit v1.2.0).
- **Kein „Tag-Cut-off"-Modus** („ab Item N offen") — Phase 2, kommt in
  einem späteren Release falls bei der Nutzung gewünscht.
- **Keine Item-IDs in den Trip-Daten** — Items werden via Index
  identifiziert. Wenn Trip-Code refactored wird (Items umgestellt),
  könnte State leicht verschoben sein. In der Praxis selten relevant.
- **Stats-Badge zeigt nur done/skipped, keine Notes**. Notizen sind
  sichtbar direkt unter dem Item.
