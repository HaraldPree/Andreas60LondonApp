# v1.15.0 — Print-Export-Abstraktion (PrintProvider Interface)

**Datum**: 26.05.2026
**Status**: deployed
**Git Tag**: v1.15.0

## Anlass

hp+ Hybrid-Strategie Phase 1 braucht eine vertrieblich verkaufbare
**Print-Komponente** — sei es klassisches Foto-Buch nach Reise
(Polarsteps-Modell) oder Programm-Heft vor Abreise
(Reiseführer-Ersatz-Modell, vermutlicher Pivot 25.05.2026).

Welcher Anbieter konkret (HappyFoto Freistadt vs. CEWE Oldenburg vs.
Saal Digital) hängt vom Partnervertrag ab — und welches Produkt vom
Demografie-Test bei RCMK-Kunden. Beide Entscheidungen stehen aus.

**Architektur-Vorbedingung**: bevor wir das erste Mal einen echten
Anbieter anbinden, brauchen wir eine **Provider-Abstraktion**. Sonst
ist jeder Provider-Wechsel ein Refactor durch den UI-Code.

Diese Iteration baut die Abstraktion — **kein User-sichtbares
Feature**. Heute liefert nur der Mock-Provider (=  bestehender
ZIP-Export für 3rd-Party-Designer), 3 Stubs für HappyFoto/CEWE/Saal
sind angelegt und werfen bei Aufruf einen klaren
`PrintProviderUnavailableError`.

## Was ist neu (alles in `src/lib/print/`)

### `types.ts` (~150 Zeilen)

Public Types der Print-Schicht:

| Type | Zweck |
|---|---|
| `PrintProductType` | `"photoBook" \| "programGuide" \| "postcardSet"` |
| `PrintFormat` | A5/A4/30×30/20×28/13×18 |
| `PrintCoverStyle` | hardcover/softcover/lay-flat |
| `PrintSpec` | Was an den Provider rein geht (Trip, Photos, Format, …) |
| `PrintOrder` | Was zurückkommt (orderId, status, localAsset/trackingUrl) |
| `PrintOrderStatus` | draft / ready-for-download / submitted / in-production / shipped / delivered / failed |
| `PrintPreview` | Vorschau-Daten (Bilder, Seitenzahl, Preis) |
| `ProductCatalog` + `ProductEntry` | Was ein Provider kann |
| `PrintOrderOptions` | Aufruf-Optionen (Progress-Callback) |
| `PrintProviderUnavailableError` | Wird geworfen wenn Stub / nicht unterstützt |

### `PrintProvider.ts`

Das Interface, das jeder Provider implementieren muss:

```ts
interface PrintProvider {
  readonly id: string;
  readonly name: string;
  readonly description: string;
  listProducts(): Promise<ProductCatalog>;
  createPreview(spec: PrintSpec): Promise<PrintPreview>;
  createOrder(spec: PrintSpec, opts?: PrintOrderOptions): Promise<PrintOrder>;
  getOrderStatus(orderId: string): Promise<PrintOrder>;
}
```

### `registry.ts`

Zentrale Provider-Auswahl:
- `getActivePrintProvider()` — liefert den per env-Variable
  (`NEXT_PUBLIC_PRINT_PROVIDER`) konfigurierten Provider, Default `mock`
- `getPrintProvider(id)` — gezielter Lookup
- `listAvailableProviders()` — alle registrierten, aktiver Provider zuerst

Künftige Multi-Tenant-Variante (Phase 2) lässt diese Funktion pro
Tenant entscheiden — App-Code bleibt unangetastet.

### `providers/`

| Datei | Provider | Status |
|---|---|---|
| `mockProvider.ts` | Lokaler ZIP-Export (3rd-Party-Designer) | **Aktiv** — wraps bestehendes `buildPhotoBookZip()` 1:1 |
| `happyFotoProvider.ts` | HappyFoto Freistadt (Pilot) | Stub — wirft `PrintProviderUnavailableError` |
| `cewProvider.ts` | CEWE Oldenburg (Rollout) | Stub |
| `saalProvider.ts` | Saal Digital (Premium / Phase 3) | Stub |

Jeder Stub hat einen ausführlichen JSDoc-Block mit:
- strategischer Begründung (warum dieser Provider)
- konkretem To-Do für die spätere Aktivierung (API, Vertrag, Format-Mapping)

### Aufrufer angepasst: `src/components/photos/PhotoBookExportButton.tsx`

Statt direktem `buildPhotoBookZip()`-Import jetzt über
`getActivePrintProvider().createOrder(...)`. Verhalten 1:1 identisch
weil Default-Provider Mock ist und intern dieselbe Pipeline ruft.

Bonus: Progress-Callback wird über das neue `PrintOrderOptions`-Interface
durchgereicht — Provider-übergreifend wiederverwendbar.

## Was funktioniert NICHT (bewusst nicht jetzt)

- **HappyFoto/CEWE/Saal echte API-Calls** — warten auf Partnervertrag
  + API-Keys + Strategie-Entscheidung. Stubs sind angelegt.
- **UI Provider-Picker** — heute gibt's nur einen aktiven Provider.
  Sobald ≥2 aktiv sind, kommt eine UI-Auswahl.
- **`programGuide`-Produkt** (Reiseführer-Heft vor Reise) — Type ist
  vorgesehen, kein Renderer existiert. Kommt mit dem Strategie-Pivot
  falls Reisebüros das Format wollen.
- **`postcardSet`** — analog vorbereitet, kein Renderer.
- **Order-Persistierung** — Mock-Orders haben keine Historie. Sobald
  remote Provider live sind, kommt ein localStorage- oder
  Server-Cache.

## Strategische Einordnung

- **Phase 1**: Mock bleibt aktiv. Sobald HappyFoto-Vertrag steht,
  Aktivierung via env-Variable + `happyFotoProvider`-Body
  implementieren. UI bleibt unangetastet.
- **Phase 2**: CEWE als Rollout-Provider für skalierte Vermarktung.
  Multi-Tenant: pro Reisebüro konfigurierbarer Default-Provider.
- **Phase 3**: Saal Digital als Premium-Anker für B2C-Substitut-Marke.
