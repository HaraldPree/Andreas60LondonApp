# v1.10.0 — AI-Pre-Trip-Event-Recherche

**Datum**: 23.05.2026
**Status**: deployed
**Git Tag**: v1.10.0

## Anlass

v1.9.0 hat die Datenstruktur für Events geschaffen, aber die Pflege ist
manuell — pro Stadt eine `.ts`-Datei mit händisch recherchierten Events.
Skaliert nicht. v1.10.0 fügt die **AI-Recherche** dazu: Claude bekommt
Stadt + Datum-Range und schlägt verifizierbare Events vor.

## Was ist neu

### 1. Anti-Halluzinations-strenger Recherche-Prompt (`src/lib/eventResearchPrompt.ts`)

Härter als der Companion-Prompt, weil hier **neue Daten generiert** werden
(nicht nur Trip-Daten paraphrasiert). Kernregeln:

- Jedes Event MUSS eine offizielle URL als Quelle haben
- Bei Datums-Unsicherheit → nicht aufnehmen
- Generische Platzhalter („verschiedene Konzerte im Hyde Park") → verboten
- Output strikt als JSON-Block (kein Freitext drumherum)
- `confidence: "low"` → wird server-seitig verworfen

### 2. API-Route (`src/app/api/research/events`)

`POST /api/research/events` mit Body:

```ts
{
  city: string,                 // z.B. "London"
  fromDate: string,             // ISO YYYY-MM-DD
  toDate: string,               // ISO YYYY-MM-DD
  existingEventIds?: string[],  // zur Duplikat-Vermeidung
  context?: string              // z.B. "60. Geburtstag, 5 Erwachsene"
}
```

**Server-seitige Anti-Halluzinations-Filter** (zusätzlich zum Prompt):
- Events ohne `source` (min 5 Zeichen) → verworfen
- Events ohne gültiges `startDate`/`endDate` → verworfen
- Events mit `confidence: "low"` → verworfen
- Category-Whitelist (10 Werte) → ungültig wird zu `"other"` normalisiert
- Recurring-Whitelist → ungültig wird zu `null`
- `lastVerified` wird **server-seitig** auf heute gesetzt (Claude darf das nicht lügen)

**Response**:

```ts
{
  events: ResearchEvent[],      // verifizierte Vorschläge
  notes?: string,               // Claudes Recherche-Hinweise
  filteredCount?: number,       // wie viele verworfen wurden
  filterReasons?: string[]      // warum
}
```

**Limits**:
- Body max 10 KB
- city max 80 Zeichen
- context max 500 Zeichen
- maxDuration 60 s (Recherche dauert)
- 10-Minuten-Client-Cache

### 3. Research-UI (`/research/events?trip=…`)

**PIN-gated** (nicht in `PUBLIC_PATH_PREFIXES`). Dezent platziert:
nur über `<details>`-Block „Veranstalter-Tools" auf der Landing-Page
erreichbar — bewusst nicht im normalen Reise-Tab-Flow, weil das eine
**Admin-Funktion für Reise-Vorbereiter** ist, nicht für Mitreisende.

**Form** (alles aus Trip vorausgefüllt wenn `?trip=` gesetzt):
- Stadt
- Von- und Bis-Datum
- Kontext (optional, hilft beim Kuratieren)
- Anzeige: „N bekannte Event-IDs werden zur Duplikat-Vermeidung übergeben"

**Ergebnis-Anzeige**:
- Erfolgs-Banner: „N Events verifiziert · M verworfen"
- Pro Event eine Card mit:
  - Name + Kategorie + Confidence-Pill (verified/prüfen)
  - Beschreibung, Location, Datum, Cost
  - Visitor-Tipps
  - „Quelle prüfen"-Link (öffnet externen Veranstalter)
  - **„Als TS kopieren"-Button** → schreibt ein einzeln-einklebbares
    Snippet ins Clipboard
- **„Alle als TS-Array kopieren"-Button** oben → kompletter Block für die
  `[city]-events.ts`-Datei
- Filter-Reasons als ausklappbares `<details>` (Transparenz: was wurde verworfen?)

### 4. Diagnose-Block bei Parse-Fehler

Wenn Claude keinen `\`\`\`json`-Block liefert, wird der Rohtext in einer
`<pre>`-Box angezeigt — damit der Operator sieht was schiefging und
ggf. den Prompt nachschärfen kann.

## Workflow für Harald

```
1. Vor jeder neuen Reise: Landing-Page → "Veranstalter-Tools" → Event-Recherche
2. City + Daten vorausgefüllt (wenn Trip-Slug schon angelegt)
3. "Recherchieren" klicken (10-30 s Wartezeit)
4. Vorschläge prüfen — jeder Source-Link wird geklickt, jeder Termin
   gegen offizielle Site verifiziert
5. Passende Events einzeln via "Als TS kopieren" ODER alle als
   Array-Block → fügt in src/data/trips/[city]-events.ts ein
6. Commit + Push wie gewohnt
```

## Was funktioniert NICHT (bekannte Limits)

- **Kein Web-Search-Tool aktiv**: Claude greift nur auf Trainings-Daten zu
  (Stand ~Anfang 2025). Annual-Events stimmen meist, taggenaue One-Offs
  weniger. → kommt in v1.10.1 wenn nötig
- **Kein Auto-Save** in Trip-Dateien: User-Approval-Pflicht via Copy-Paste.
  Bewusst so — Anti-Halluzination + Code-Review-Disziplin
- **Pro Recherche ~1-3 € Claude-Kosten** (~4k Output-Tokens): nicht pro User,
  sondern pro Reise-Vorbereitung. Cost-Risiko niedrig
- **Keine Persistenz** der Recherche-Ergebnisse: bei Refresh sind sie weg.
  Snippet-Kopieren ist der einzige Output-Kanal
- **Pro Recherche-Call ein einzelner Request**: kein Tool-Loop, kein
  Streaming. Recherche dauert 10-30 s + Resultat in einem Rutsch

## Architektur-Notizen

- API-Route hat eigene Validierungs-Pipeline (`validateAndNormalize`) statt
  Zod o.ä. — minimal, kein neuer Dependency
- `lastVerified` wird server-seitig gesetzt — Claude darf das Datum
  nicht lügen
- TS-Snippet-Builder im Client (`buildEventSnippet`) — entspricht exakt
  dem Format aus `london-2026-events.ts`, inkl. `VERIFIED`-Konstante
  als Referenz (statt Datum hardcoded)

## Migrations-Hinweise

- **Keine Schema-Migration** — pures additives Feature
- **Keine neuen Env-Vars** — nutzt vorhandenen `APP_ANTHROPIC_KEY`
- **PIN-Gate** schützt die Route automatisch (`/research/*` nicht in
  `PUBLIC_PATH_PREFIXES`)
- App-Reload nötig (Cache-Bust)
