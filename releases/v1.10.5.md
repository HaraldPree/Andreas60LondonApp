# v1.10.5 — Drei Save-Pfade nebeneinander (Samsung-Share-Sheet bug umgangen)

**Datum**: 23.05.2026
**Status**: deployed
**Git Tag**: v1.10.5

## Anlass

Harald-Bug nach v1.10.4:

> „jetzt funktioniert das herunterladen wieder nicht, bilder einlesen
> usw geht dann steht pdf speichern pdf bereit und auf Handy Teilen
> sheet öffnet sich, das ist aber nicht da"

Samsung Internet meldet `canShare({files: [...]})` = `true`, dann
`navigator.share(...)` läuft durch **ohne Sheet zu zeigen und ohne Fehler
zu werfen**. Klassischer Browser-Bug — User sieht: nichts.

## Lösung

Statt einem einzigen Save-Pfad mit Fallback-Kaskade gibt's jetzt **drei
sichtbare Buttons**, vom besten zum letzten:

1. 🟦 **„PDF speichern (Teilen-Sheet)"** — primärer Versuch via `navigator.share`
   - **WICHTIG**: ohne vorherigen `canShare`-Check (Samsung lügt)
   - Bei Erfolg: Sheet öffnet sich, User wählt App/Speichern
2. 🟨 **„PDF in neuem Tab öffnen"** — `window.open(blob:url, "_blank")`
   - PDF rendert inline im Browser
   - User kann via 3-Punkte-Menü → „Seite speichern" sichern
3. 🩶 **„Direct-Download (Desktop / Chrome)"** — klassischer `<a download>`
   - Funktioniert nur auf Desktop + Chrome Android
   - Samsung Internet macht hier about:blank — explizit als Letzte-Option
     gekennzeichnet

Plus: `saveStatus`-Feedback nach dem Klick zeigt Hinweis-Text was passiert
ist („PDF in neuem Tab geöffnet — dort speichern").

## Code-Änderungen

### handleSave — robuste 3-Stufen-Kaskade

```ts
const handleSave = async () => {
  setSaveStatus("trying");

  // 1) Share-API ohne canShare
  if (navigator.share && File) {
    try {
      await navigator.share({ files: [file], title: filename });
      return;
    } catch (e) {
      if (/abort|cancel/.test(e.message)) return;
    }
  }

  // 2) window.open in Tab
  if (window.open(url, "_blank")) {
    setSaveStatus("opened-tab");
    return;
  }

  // 3) Anchor-Click
  const a = document.createElement("a");
  a.href = url; a.download = filename;
  a.click();
  setSaveStatus("anchor-click");
};
```

### handleOpenInTab — separater Button für „Plan B"

Gleiche Logik wie Schritt 2 oben, aber direkt zugänglich falls Hauptbutton
versagt ohne Feedback (Samsung-Stillschweigen-Bug).

### UI — drei sichtbare Optionen statt einer

Vorher: 1 Save-Button mit verstecktem Fallback. Wenn nichts passiert →
User weiß nicht warum.

Jetzt: 3 Buttons in absteigender Größe + Wichtigkeit:
- Blau-prominent
- Gold-mittel
- Cream-klein

Plus dynamischer Status-Text unten.
